### Relational database ###
    * Also called as SQL DB
    * These DB use a relational model to organise data into table with rows and column 
    * Data are stored in tables
    * Tables are often related
    * It is a row based DB because, one each information is stored in the form of row
    * Usually used for websites/apps

    Eg: MySQL, PostgreSQL, Oracle, SQL Server, Microsoft access


### NoSQL database ###
    * Not only SQL
    * These DB are used to handle large amount of unstructured, semi-structured data like photos, videos, pdfs etc
    * such data are usually present in Social Media Application

    Eg: MongoDB


### Column database ###
    * These DBs store data in columns unlike relational DBs which stores row wise. 
    * It is helpful for data warehousing and analytical application
    * It is comparatively fast
    * Usually used for analysis

    Eg: Amazon Redshift, Google Query

    Eg:
        one table for all student names
        one table for all student branch
        one table for all student cgpa

        how is it stored in memory?
        -   name1, name2, ... , branch1, branch2, ... , cpga1, cgpa2, ...


### Graph database ###
    * Databases are used to store and query graph-structured data
    * Usually used in social media, networking kind of applications, recommendation system etc
    * That is, it helps to store complex relationship data

    Eg: Neo4j, Amazon Neptune


### Key-Value database ###
    * They store data in key - value pairs
    * Useful in caching and simple data storing application

    Eg: Redis, Amazon DynamoDB

### KEYS ###
    * A key is an attribute/column or set of attribute, that uniquely identifies a Row in the table
    Eg: 
    In student table with columns - roll num, name, branch, email,
    EMAIL and ROLL NUM can become the KEY

    * It plays crucial roles in ensuring the integrity and reliability of the DB


    # Types of Keys

    1. Super Key
        * It is a combination of columns that uniquely identifies any row within RDBMS table

        Eg: 
        In student table with columns - roll num, name, branch, email,
        All the following (uniquely identifies a row) becomes Super Key
            - roll num
            - email
            - roll num + name
            - roll num + branch
            - roll num + email
            - email + branch
            - email + name
            - name + branch + email
            - roll num + branch + name
            - roll num + name + email
            - roll num + branch + email
            - all 4 columns

    2. Candidate Key
        * A candidate key is a minimal Super key, meaning it has no redundant attributes.
        * It is the smallest set of attributes that can be used to uniquely identify a row in the table

        Eg: From the above super keys, only "roll num" and "email" are the candidate keys, because rest all keys will be redundant since they are a combination of roll num or email or both

    3. Primary Key
        * Becomes the main key of the table
        * It is the unique identifier for a row
        * It can't be NULL
        * One table contains ONLY ONE PRIMARY KEY
        * It can't be duplicate

        Mandatory criteria - not a null, can't be duplicate
        Good to have - numerical, small, constant

        Eg: "Roll Num" becomes the Primary Key to uniquely identify a row

    4. Alternate Key
        * It is a candidate key which is not a Primary Key
        * Alternate key = candidate key - primary key
        Eg: email

    5. Composite Key
        * It is a primary key made of TWO or MORE attribute

    6. Surrogate Key
        * Table with columns:  name, branch, cgpa 
            - none of the column or combination of columns can't become Primary Key
            - Adding a new column: student_id which acts as a primary key
            - student_id is the surrogate key

    7. Foreign Key
        * It is a primary key from one table used to establish the relationship with other table
        * Primary key of one table is called as foreign key in another table when used in the second table




### Cardinality of the Relationships ###
    * It refers to the number of occurrences of an entity in a relationship with another entity
    * Entity is something which can be represented in terms of Table - student, restaurant etc
    * Cardinality defines the number of instances of one entity that can be associated with a single instance of the related entity

    It is of three types 
    1. One to One relationship
        * Two entities has one to one relationship
        eg: person and a driving relationship - each person has only one DL, one DL belongs to only one person

        ** Only one table is required to store the entire info **

    2. One to Many relationship
        * student and branch - a student belongs to only branch, but a branch contains many students

        ** Two table is required to store the entire info **

    3. Many to Many relationship
        * student and courses - one student can enroll in multiple courses and a single course can be enrolled by multiple students
        * restaurant and order
        * restaurant and menu

        ** This requires three tables to store the entire data **

### Referential Actions - If 2 tables are related via foreign key, what happens to one table upon deleting or updating a key in another table.

It is of 4 types
    1. Restrict - default behaviour
    2. Cascade
    3. Set null
    4. Set default


Consider the following schema:
* Two Table - customers and orders
* Customers Table -> customer_id(pk), name, email
* Orders Table -> order_id(pk), customer_id(foreignkey), date


1. Restrict : 
    * if there is a customer with id: 4 and if his id is also present in orders, then the DBMS doesn't allow to delete/modify that customer(with id: 4) from the customer table.
    * This is the default behaviour

2. Cascade : 
    * if there is a customer with id: 4 and if his id is also present in orders, then any changes will be reflected in the orders table also. If that customer is deleted from the customer table, then all the entries related to his customer id will also be deleted from the orders table.
    
3. Set null : 
    * upon deleting that customer in the customer table, the foreign key in the orders table will be set to null

4. Set default: 
    * upon deleting that customer in the customer table, the foreignkey in the orders table will be set to the default value

#####################################################################################################

* It is a way to combine data from two or more tables based on a common connecting column
* It is useful when data is distributed accross tables. Joins creates a temporary table on which the operations can be performed

Why do we need multiple tables?
1. To reduce data redundancy
2. Reduces update/delete anomoly
3. Data will be well organised  

The entire process is called as Normalization - unique entities are divided into different tables




########### Types Of Joins ###########
6 Types of Joins - inner, left, right, full outer, cross, self


### Cross Join.
1. Also known Cartesian Product
2. Return Cartesian product of two tables being joined
3. It return all the possible combination of Rows from 2 tables
4. Applicable on rows
5. Not commonly used


### Inner Join.
1. Combines data from two or more tables based on specified condition
2. Only the rows from both the table will be returned that satisfies the specified condition
3. Gives the intersection of data on a specified condition
4. Joins on a common column


### Left Join.
1. Also called as Left Outer Join
2. Returns all the rows from left table and also the matching rows from the right table
3. If there are no matching rows in right table, the result will have null values in the columns that come from the right table
4. Joins 2 tables based on common column - it returns all the data from the left table even if there is no match in right table


### Right Join.
1. Also called as Right Outer Join
2. Returns all the rows from right table and also the matching rows from the left table
3. If there are no matching rows in left table, the result will have null values in the columns that come from the left table
4. Joins 2 tables based on common column - it returns all the data from the right table even if there is no match in left table


### Full Outer Join.
1. Also called as Full Join
2. Returns all matching rows from both left and right table and also non matching rows from both tables.
3. It fills NULL values for columns that don't have a match


### Self Join.
1. A table is joined with itself
2. The same table is treated as two different tables
3. Used when one need compare a row with another row if the same table
